# 简历描述参考

项目名称：618 高并发秒杀与抽奖平台

项目描述：

基于 Spring Cloud Alibaba 搭建的微服务秒杀系统，模拟 618 大促 100000 台手机发放场景，其中 90000 台走秒杀链路，10000 台走抽奖链路。系统使用 Redis Lua 承接秒杀高并发库存扣减，RabbitMQ 异步削峰完成订单落库，MySQL 保证最终库存与订单数据一致，Nacos 负责服务注册发现，Gateway 统一鉴权、路由和静态资源托管，并提供可直接通过 `http://localhost:8080` 访问的原生前端演示控制台。

可写亮点：

- 设计 Gateway + JWT 统一认证方案，登录后网关解析 token 并透传用户上下文，业务服务通过 ThreadLocal 获取当前用户。
- 秒杀入口使用 Redis Lua 脚本原子完成库存扣减、用户去重、一人一单校验，将热点库存访问从 MySQL 前移到 Redis。
- 使用 RabbitMQ 解耦抢购请求和订单落库，订单服务消费消息时通过 `request_id` 和数据库唯一索引实现幂等。
- 抽奖报名使用 Redis Set 去重，20:45 定时任务随机抽取中奖用户并批量投递订单消息，支持中奖结果查询。
- 商品服务维护活动库存权威数据，Redis 预扣减负责高并发入口，MySQL 扣减负责最终一致性校准。
- 将前端从本地 `file://` 页面升级为 Gateway 托管的静态控制台，用户直接访问 `http://localhost:8080` 即可完成登录、浏览活动、秒杀、抽奖和订单查询。
- 重构原生前端代码，将单体 `app.js` 拆分为 `core`、`api`、`pages` 模块，封装 hash router、请求层、会话状态、通用组件和页面渲染逻辑，降低页面扩展成本。
- 新增独立登录页和会话联动机制，支持 demo/admin 快速登录、注册、JWT 本地保存、401 自动清理会话和页面登录态刷新。
- 完善前端演示体验，新增商品中心、活动中心、秒杀入口、抽奖中心、我的订单等页面，并提供库存进度、倒计时、轮盘/棋盘抽奖动效、中奖播报和异步订单结果展示。
- 使用 Docker Compose 编排 MySQL、Redis、RabbitMQ、Nacos，提供 SQL 初始化、API 示例、启动脚本和压测脚本，便于本地演示和 GitHub 展示。

面试可讲：

- 为什么不用数据库直接扣库存：MySQL 行锁在大促高峰下会成为瓶颈，Redis Lua 能在单线程模型中保证原子性并显著降低数据库压力。
- 为什么要用 MQ：抢购结果可以先排队，订单落库异步执行，削平瞬时流量峰值。
- 如何防止重复下单：入口层 Redis Set 拦截重复请求，订单层 `request_id` 唯一索引兜底。
- 如何处理异常补偿：秒杀 MQ 投递失败会执行 Redis 回滚脚本；订单落库失败会调用商品服务释放 MySQL 库存。
- 为什么将前端放到 Gateway：避免本地 `file://` 访问带来的演示割裂和跨域配置问题，统一入口更接近真实系统访问路径。
- 前端为什么做模块化：把路由、状态、请求、组件和页面拆开后，新增商品页、活动页、运维页等功能不需要继续堆叠单文件脚本，演示代码更容易维护。
