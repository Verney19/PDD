# 简历描述参考

项目名称：618 高并发秒杀与抽奖平台

项目描述：

基于 Spring Cloud Alibaba 搭建的微服务秒杀系统，模拟 618 大促 100000 台手机发放场景，其中 90000 台走秒杀链路，10000 台走抽奖链路。系统使用 Redis Lua 承接秒杀高并发库存扣减，RabbitMQ 异步削峰完成订单落库，MySQL 保证最终库存与订单数据一致，Nacos 负责服务注册发现，Gateway 统一鉴权和路由。

可写亮点：

- 设计 Gateway + JWT 统一认证方案，登录后网关解析 token 并透传用户上下文，业务服务通过 ThreadLocal 获取当前用户。
- 秒杀入口使用 Redis Lua 脚本原子完成库存扣减、用户去重、一人一单校验，将热点库存访问从 MySQL 前移到 Redis。
- 使用 RabbitMQ 解耦抢购请求和订单落库，订单服务消费消息时通过 `request_id` 和数据库唯一索引实现幂等。
- 抽奖报名使用 Redis Set 去重，20:45 定时任务随机抽取中奖用户并批量投递订单消息，支持中奖结果查询。
- 商品服务维护活动库存权威数据，Redis 预扣减负责高并发入口，MySQL 扣减负责最终一致性校准。
- 使用 Docker Compose 编排 MySQL、Redis、RabbitMQ、Nacos，提供 SQL 初始化、API 示例和压测脚本，便于本地演示和 GitHub 展示。

面试可讲：

- 为什么不用数据库直接扣库存：MySQL 行锁在大促高峰下会成为瓶颈，Redis Lua 能在单线程模型中保证原子性并显著降低数据库压力。
- 为什么要用 MQ：抢购结果可以先排队，订单落库异步执行，削平瞬时流量峰值。
- 如何防止重复下单：入口层 Redis Set 拦截重复请求，订单层 `request_id` 唯一索引兜底。
- 如何处理异常补偿：秒杀 MQ 投递失败会执行 Redis 回滚脚本；订单落库失败会调用商品服务释放 MySQL 库存。
